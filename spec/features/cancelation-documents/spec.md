# cancelation-documents — спецификация

## Goal

Дать админу LMS при редактировании участника возможность подтвердить документы профиля или аннулировать их с комментарием, с удалением файлов из хранилища и уведомлением пользователя по email; после подтверждения участник не может менять документы самостоятельно.

## In-scope

- Статусы жизненного цикла документа (минимум: ожидает проверки / подтверждён / аннулирован) и сохранение комментария админа при аннулировании.
- Действия в админке: «Подтвердить» и «Аннулировать» (с обязательным комментарием при аннулировании).
- Удаление файла из физического storage при аннулировании и согласованная очистка записей (`lms_profile_documents`, при необходимости `uploaded_media` / связанные сущности — по факту кода).
- Отправка email пользователю с текстом комментария админа (очередь/синхронно — по принятому в проекте паттерну).
- На стороне участника: после подтверждения — запрет самостоятельной замены документов (только через поддержку; явное сообщение в UI).
- После аннулирования — возможность повторной загрузки документов тем же типом.

## Out-of-scope

- Отдельный тикетинг/support-инструмент; достаточно текста «обратитесь в поддержку» без интеграции.
- Полный аудит `UploadedMedia` vs `Media` (см. открытый вопрос в `/spec/90-open-questions.md`); в рамках фичи — только согласованное удаление для путей, используемых документами профиля.
- ~~**Фаза 2:** заявка на замену подтверждённого документа~~ — реализовано: участник «Запросить замену» + комментарий; админ одобряет (файл снимается, статус снова «на проверке») или отклоняет (комментарий админа без email в v1).

## Constraints

- Команды и проверки — только через Docker по паттерну из `spec-continuation` (`source docker/.env.*`, `docker exec ${APP_NAME}_fpm`).
- Переиспользовать существующие UI-примитивы (`resources/js/components/ui/`, shared, composables: `useToast`, `useConfirm`, `Dialog` и т.д.).
- Не ломать текущие загрузки и маршруты без миграции данных для существующих строк `lms_profile_documents`.

## Open questions

- Нет блокирующих. Реализация: диск — `config('filesystems.upload_disk')`; `UploadedMedia` не используется для документов профиля LMS.

## Реализованное поведение (кратко)

- Колонки `status`, `admin_comment`, `reviewed_at`; при аннулировании файл удаляется из storage, `file_path`/`original_name` сбрасываются в пустую строку, запись по типу сохраняется для комментария.
