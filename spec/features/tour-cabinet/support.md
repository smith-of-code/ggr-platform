# Обращения в поддержку через ЛК туров

Дополняет `spec.md` раздела «Личный кабинет туров». Реализация — отдельная итерация (миграции, маршруты, UI ЛК и админки).

## Решения по продукту (зафиксировано)

| Вопрос | Решение |
|--------|---------|
| Кто обрабатывает обращения | **Только админы портала** (`users.is_admin`). Отдельная роль «поддержка» с ограниченными правами в MVP не вводим. |
| Вложения в MVP | **Да.** Прикрепление файлов к сообщениям тикета (см. ограничения ниже). |
| Канал email (например tours@…) | **Остаётся** параллельно тикетам: на сайте/в ЛК явно можно оставить текст «или напишите на …». Тикеты — основной структурированный канал внутри ЛК; почта — ручной альтернативный контакт, **без** автоматической связи с тикетом и **без** авто-рассылки из приложения (см. ниже). |
| Почтовые уведомления из приложения | **Сейчас не используются** — только ЛК участника и админка; при необходимости включить позже (язык шаблонов тогда не фиксировать «только RU»). |
| Видимость раздела «Поддержка» | Показывать всем, кто имеет доступ к **`/tour-cabinet`** по текущим правилам middleware: в т.ч. участникам **без** `is_tour_cabinet_user`, но **с** записью `lms_profiles` (как и остальной ЛК туров). |

## Назначение

- Отдельный поток от **заявок на тур** (`applications`, `type = tour`): вопросы по конкурсу, технике, статусам и пр.
- История переписки в ЛК и ответы из админки (без отправки писем из приложения в текущей версии).

## Модель данных (целевая)

1. **`tour_cabinet_support_tickets`** (или согласованное имя): `user_id`, `subject`, `category` (enum/string), `status` (`open`, `pending_user`, `resolved`, `closed` — уточнить набор при реализации), `created_at`, `updated_at`, опционально `last_message_at` для сортировки очереди.
2. **`tour_cabinet_support_messages`**: `ticket_id`, автор (`user` / `admin`), `user_id` или `admin_user_id`, `body` (text), `created_at`.
3. **Вложения:** отдельная таблица **`tour_cabinet_support_attachments`** (или полиморф к сообщению): `message_id`, `disk`, `path`, `original_filename`, `mime_type`, `size_bytes`, `created_at`. Хранение — тот же подход, что и для других upload-ов портала (`filesystems.upload_disk`), с лимитами по размеру и MIME (whitelist: изображения, PDF, типичные офисные — зафиксировать в коде/валидации).

**Рекомендация по содержимому:** в UI и в короткой подсказке у формы указать, что **документы уровня паспорта** лучше не загружать в общий тикет; для паспортных данных — отдельный защищённый процесс (когда появится). На MVP технически вложения включены, политика — предупреждение + разумный whitelist MIME.

## ЛК участника (`/tour-cabinet`)

- Маршруты: список тикетов, создание, просмотр треда, загрузка вложений к **новому сообщению** (и при создании тикета — первое сообщение + вложения).
- Rate limiting на создание тикетов и на загрузку файлов.
- Ссылка/блок «Поддержка» доступен всем с доступом к ЛК туров (см. таблицу выше).

## Админка портала

- Раздел только для **`is_admin`**: очередь тикетов, фильтры, карточка тикета, ответ (новое сообщение от админа), смена статуса.
- Аудит: в сообщении с `author_type = admin` хранить `user_id` админа.

## Канал tours@… (и аналоги)

- Контакт в футере/на странице помощи **сохраняется**.
- В письмах с почты тикеты **автоматически не создаются**; при дублировании сценария оператор может вручную завести тикет в админке (опционально позже — не MVP).

## Связь с основным spec ЛК туров

Маршруты и навигация зафиксированы в `spec.md` (таблица ЛК и блок админки). В дашборде ЛК — ссылка «Поддержка»; в админ-сайдбаре — «Обращения ЛК туров».
