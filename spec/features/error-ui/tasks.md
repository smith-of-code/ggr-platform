# error-ui — задачи

## 1. Аудит payload событий Inertia

- **Goal**: Понять, какие поля доступны в `router.on('invalid')` и `router.on('error')` в текущей версии `@inertiajs/vue3` и что реально возвращает Laravel при 422/500.
- **Scope**: `resources/js/app.js`, документация/типы Inertia в `node_modules` (только чтение), при необходимости один тестовый ответ из feature-теста или ручной запрос.
- **DoD**: Краткая заметка в комментарии или в коде: откуда читать `response` / тело и как парсить JSON.
- **Verify**: Не требуется сборка; при необходимости `npm run build` по паттерну из `spec-continuation`.

## 2. Утилита `formatServerError` (или аналог)

- **Goal**: Одна функция: на вход ответ/текст/объект ошибок — на выход строка или массив строк для toast.
- **Scope**: новый файл рядом с composables, например `resources/js/utils/inertiaErrors.js` (или `.ts` по конвенции проекта).
- **DoD**: Покрыты случаи: JSON `{ message }`, `{ errors: { field: ['msg'] } }`, не-JSON тело, пустой ответ → осмысленный fallback на русском.
- **Verify**: `source docker/.env.local && docker exec ${APP_NAME}_fpm npm run build`.

## 3. Обработчик `invalid`

- **Goal**: Вместо фиксированного «Сервер вернул некорректный ответ» показывать результат утилиты из п.2.
- **Scope**: `resources/js/app.js`.
- **DoD**: При типичном JSON с `message` пользователь видит этот текст; при validation — сводка по правилам п.2.
- **Verify**: `npm run build` в контейнере `fpm` по паттерну `spec-continuation`.

## 4. Обработчик `error`

- **Goal**: Если в событии есть детализируемая причина — показать её, иначе оставить или слегка уточнить общее сообщение.
- **Scope**: `resources/js/app.js`.
- **DoD**: Нет регрессии для сетевых сбоев без тела ответа.
- **Verify**: `npm run build` в контейнере `fpm`.

## 5. Ветка `finish` и `props.errors`

- **Goal**: Заменить единственное «Пожалуйста, исправьте ошибки в форме» на краткую человекочитаемую сводку (например перечисление первых сообщений или одно главное).
- **Scope**: `resources/js/app.js`, при необходимости вызов той же утилиты для объекта `errors` из props.
- **DoD**: При нескольких полях сообщение информативно; при одном — показывается текст этой ошибки.
- **Verify**: Ручная проверка формы с ошибками + `npm run build` в контейнере `fpm`.

## 6. `flash.error` и отсутствие дублирования

- **Goal**: Сохранить показ `flash.error`; не показывать два toast за один цикл, если и flash, и errors заполнены (политика: приоритет или объединение — зафиксировать в коде).
- **Scope**: `resources/js/app.js`.
- **DoD**: Поведение документировано комментарием; нет двойных одинаковых toast.
- **Verify**: Ручная проверка + `npm run build` в контейнере `fpm`.

## 7. Ограничение длины и количества строк

- **Goal**: Обрезка/лимит N сообщений валидации в toast согласно `spec.md` Constraints.
- **Scope**: утилита из п.2, при необходимости константы.
- **DoD**: Массовые `errors` не заполняют экран бесконечным текстом.
- **Verify**: `npm run build` в контейнере `fpm`.

## 8. Регрессия и финальная верификация

- **Goal**: Пройти сценарии: успешный visit, 422 форма, `invalid` (не-Inertia HTML), опционально 500 с JSON `message`.
- **Scope**: весь изменённый JS; при необходимости минимальный Pest/feature-тест только если в проекте уже тестируют Inertia сценарии (не обязательно).
- **DoD**: Нет необработанных исключений в консоли при стандартных сценариях.
- **Verify**: `npm run build` в контейнере `fpm`; при наличии e2e — не расширять scope без необходимости.
