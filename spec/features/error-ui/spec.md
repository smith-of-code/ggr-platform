# error-ui — Человекочитаемые ошибки Inertia/сервера

## Goal

Показывать пользователю конкретный текст ошибки с сервера (включая валидацию Laravel), а не только общие фразы вроде «Сервер вернул некорректный ответ» и «Пожалуйста, исправьте ошибки в форме».

## In-scope

- Глобальные обработчики Inertia Router в `resources/js/app.js`: события `invalid`, `error`, а также ветка `finish` при наличии `page.props.errors` (validation).
- Извлечение сообщения из тела ответа, когда это JSON с полями `message`, `errors` (объект поле → массив строк) или аналогичный контракт Laravel/Inertia для 4xx/5xx.
- Агрегация ошибок валидации в одну или несколько понятных строк для toast (например первое сообщение, или краткий список без дублирования технических ключей без контекста).
- Сохранение существующего поведения для `page.props.flash.error` (показывать текст как сейчас).

## Out-of-scope

- Полная интернационализация (i18n) сообщений бэкенда.
- Перепись всех контроллеров/Form Request под единый формат сообщений — только потребление уже отдаваемых структур.
- Отдельный UI-модал для ошибок (остаётся toast через `useToast`), если явно не решат иначе в реализации.

## Constraints

- Не дублировать длинные stack trace или сырой HTML в toast.
- Ограничить длину/количество строк в toast, чтобы не ломать UX при массовых validation errors.

## Open questions

- Нет: точка входа зафиксирована в `resources/js/app.js`; формат `errors` для Inertia совпадает с типичным Laravel.

## Реализация (актуально)

- `resources/js/utils/inertiaErrors.js` — сбор сообщений валидации, разбор JSON non-Inertia ответов; лимиты длины и числа полей.
- `resources/js/app.js` — `exception` (сеть / неожиданные сбои), `error` (toast по `detail.errors`), `invalid` (toast по `detail.response`), `finish` только для `flash.error` (без второго toast по тем же ошибкам валидации).
